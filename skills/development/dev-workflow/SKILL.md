# Dev Workflow - 完整开发工作流

## 概述

dev-workflow 提供了一套系统化的六阶段开发方法论，帮助你确保开发过程的质量和可追溯性。适用于需要严格质量控制的功能开发任务。

## 何时使用

- 新功能开发
- 重大重构
- 复杂 Bug 修复
- 核心模块变更
- 多人协作项目

## 何时不使用

- 简单的文字修改
- 紧急 Hotfix
- 原型验证阶段
- 小的样式调整

## 工作流程

### 阶段 1: 需求明确

**目标**: 将模糊想法转化为清晰的需求文档

**任务清单**:
- [ ] 收集用户故事和需求
- [ ] 定义功能范围（包含/不包含）
- [ ] 确定验收标准
- [ ] 识别依赖关系
- [ ] 评估风险和假设
- [ ] 产出: 需求文档或 PRD 片段

**输出示例**:
```
## 需求: 用户密码重置功能

### 用户故事
作为用户，我希望能够重置密码，以便在忘记密码时重新登录。

### 验收标准
- [ ] 用户可以通过邮箱接收重置链接
- [ ] 重置链接有效期为1小时
- [ ] 密码重置后需要重新登录
- [ ] 发送邮件失败时有友好提示

### 范围
包含:
- 邮箱验证码发送
- 密码重置页面
- 后端 API

不包含:
- 手机验证码重置
- 安全问题验证
```

### 阶段 2: 架构设计

**目标**: 基于需求创建技术实现方案

**任务清单**:
- [ ] 设计技术方案
- [ ] 选择技术栈和库
- [ ] 设计数据模型
- [ ] 设计 API 接口
- [ ] 考虑性能和扩展性
- [ ] 画架构图（可选）
- [ ] 产出: 技术设计文档

**输出示例**:
```
## 技术设计

### 数据模型变更
```sql
ALTER TABLE users ADD COLUMN reset_token VARCHAR(255);
ALTER TABLE users ADD COLUMN reset_token_expires_at TIMESTAMP;
```

### API 设计
- POST /api/auth/forgot-password
- POST /api/auth/reset-password

### 模块结构
```
src/
├── api/
│   └── auth/
│       ├── forgot_password.py
│       └── reset_password.py
├── services/
│   └── email_service.py
└── templates/
    └── emails/
        └── password_reset.html
```
```

### 阶段 3: 任务规划

**目标**: 将设计分解为具体的编码任务

**任务清单**:
- [ ] 拆解为小的、可独立提交的任务
- [ ] 确定任务顺序和依赖
- [ ] 预估每个任务的时间
- [ ] 产出: 任务清单或 To-do 列表

**输出示例**:
```
## 任务清单

1. [ ] 创建数据库迁移 (15m)
2. [ ] 实现 EmailService.send_reset_email() (30m)
3. [ ] 实现 POST /api/auth/forgot-password (45m)
4. [ ] 添加单元测试 for forgot-password (30m)
5. [ ] 实现 POST /api/auth/reset-password (45m)
6. [ ] 添加单元测试 for reset-password (30m)
7. [ ] 创建前端密码重置页面 (1h)
8. [ ] 集成测试 (30m)
9. [ ] 文档更新 (15m)

预计总时间: ~4.5小时
```

### 阶段 4: 代码实现

**目标**: 按照任务清单逐步实现功能

**任务清单**:
- [ ] 按任务清单逐个实现
- [ ] 每个任务提交一次（小步提交）
- [ ] 写好提交信息
- [ ] 保持代码质量
- [ ] 产出: 可工作的代码

**提交信息格式**:
```
&lt;type&gt;: &lt;subject&gt;

&lt;body&gt;
```

Type: feat, fix, refactor, test, docs

### 阶段 5: 代码复核

**目标**: 严格复核代码质量、规范一致性和需求符合度

**任务清单**:
- [ ] 对照需求检查功能完整性
- [ ] 代码审查（使用 code-reviewer 技能）
- [ ] 运行测试并确保全部通过
- [ ] 检查代码规范
- [ ] 安全检查
- [ ] 产出: 复核报告

### 阶段 6: 代码优化

**目标**: 基于复核报告进行迭代改进

**任务清单**:
- [ ] 修复复核中发现的问题
- [ ] 优化性能问题
- [ ] 改进代码可读性
- [ ] 完善文档和注释
- [ ] 最终验证
- [ ] 产出: 最终高质量代码

## 核心原则

1. **先思考后编码** - 设计阶段多花时间，编码阶段更高效
2. **小步前进** - 每个提交要小而聚焦
3. **质量第一** - 不牺牲质量赶进度
4. **可追溯** - 每个决策都有记录
5. **持续验证** - 每个阶段都有验收标准

## 使用示例

### 完整流程示例

```
你：帮我开发一个用户评论功能
智能体：好的！让我们按照完整开发流程进行。

=== 阶段 1: 需求明确 ===
让我先整理需求...

✅ 需求文档已生成
请确认需求是否正确，或提出修改意见。

确认后进入阶段 2: 架构设计
```

### 快速模式示例

```
你：快速开发一个简单的 API 端点
智能体：好的，我将使用简化流程（跳过部分文档）

=== 简化开发流程 ===
1. 快速需求确认
2. 简要设计
3. 实现
4. 基础测试

开始...
```

## 最佳实践

1. **灵活调整** - 根据项目大小调整各阶段的深度
2. **及时沟通** - 有疑问立即确认，不要假设
3. **文档随行** - 文档和代码同步更新
4. **定期回顾** - 完成后总结经验教训
5. **工具辅助** - 善用 Linter、Formatter、Test 等工具

## 常见问题

### Q: 这个流程太繁琐了？

A: 流程是为了保证质量。小项目可以裁剪，大项目建议完整执行。

### Q: 需求变更了怎么办？

A: 回到阶段 1，重新评估和规划，记录变更原因。

### Q: 如何平衡速度和质量？

A: 质量是速度的基础。前期花时间，后期省时间。
