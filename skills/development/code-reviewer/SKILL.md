# Code Reviewer - 代码审查助手

## 概述

code-reviewer 是一个专业的代码审查技能，提供系统性的代码质量检查清单、最佳实践指导和常见问题识别。它可以帮助你在合并代码前发现潜在问题，提高代码库的整体质量。

## 何时使用

- 提交 Pull Request 前进行自我审查
- 团队代码审查时使用检查清单
- 学习代码最佳实践
- 重构前评估代码质量
- 新人入职代码培训

## 何时不使用

- 代码已经过完整审查时
- 需要深度架构评审时（使用专门的架构评审技能）
- 紧急 Hotfix 时

## 工作流程

### 阶段 1: 阅读与理解

- 阅读相关代码文件
- 理解代码的用途和上下文
- 识别关键模块和函数

### 阶段 2: 逐项检查

按照检查清单逐项审查：
- 代码风格与规范
- 错误处理
- 安全性
- 性能
- 可维护性

### 阶段 3: 生成报告

- 记录发现的问题
- 提供改进建议
- 给出评分和总结

## 核心原则

1. **建设性反馈** - 批评代码，不批评人
2. **具体明确** - 指出具体位置和问题
3. **提供建议** - 不仅指出问题，还要给出方案
4. **优先级排序** - 按严重程度分类问题

## 检查清单

### 代码风格

- [ ] 遵循项目的代码风格指南
- [ ] 变量和函数命名清晰有意义
- [ ] 适当的注释（不要过度注释）
- [ ] 函数长度适中（建议&lt;50行）
- [ ] 一致的缩进和格式

### 错误处理

- [ ] 适当的错误捕获和处理
- [ ] 有意义的错误信息
- [ ] 边界条件检查
- [ ] 空值/Null 检查
- [ ] 资源正确释放（文件、数据库连接等）

### 安全性

- [ ] 输入验证和清理
- [ ] SQL 注入防护（使用参数化查询）
- [ ] XSS 防护（输出编码）
- [ ] 敏感数据不记录日志
- [ ] 权限检查

### 性能

- [ ] 避免不必要的循环
- [ ] 合理的数据结构选择
- [ ] 数据库查询优化（N+1问题）
- [ ] 缓存使用合理
- [ ] 内存泄漏检查

### 可维护性

- [ ] 单一职责原则
- [ ] 模块化设计
- [ ] 可测试性（依赖注入）
- [ ] 魔法数字替换为常量
- [ ] 重复代码抽取

### 文档

- [ ] 公共 API 有文档注释
- [ ] 复杂逻辑有解释
- [ ] README 更新（如需要）
- [ ] 变更日志更新

## 使用示例

### 示例 1: 审查 Python 文件

```
你：帮我审查一下这个文件
智能体：好的！让我先阅读代码...

=== 代码审查报告 ===
文件: app/services/user_service.py

🔴 严重问题 (1)
1. 第 45 行: SQL 注入风险
   - 问题: 使用字符串拼接构建 SQL 查询
   - 建议: 使用参数化查询
   - 代码: cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")

🟡 中等问题 (2)
1. 第 23 行: 缺少错误处理
   - 问题: API 调用没有 try-except
   - 建议: 添加异常处理
2. 第 78 行: 魔法数字
   - 问题: timeout = 30
   - 建议: 定义常量 DEFAULT_TIMEOUT = 30

🟢 建议 (3)
1. 第 12 行: 函数可以更短
   - 建议: 拆分为多个小函数
2. 第 56 行: 添加类型注解
3. 第 90 行: 添加 docstring

评分: 6.5/10
建议: 修复严重问题后再合并
```

## 最佳实践

1. **小步审查** - 每次审查&lt;400行代码
2. **及时反馈** - 24小时内给出审查意见
3. **积极肯定** - 不要只找问题，也要肯定优点
4. **讨论而非命令** - 使用建议性语气
5. **自动化优先** - 能用工具检查的先让工具检查

## 常见问题

### Q: 如何处理意见分歧？

A: 以数据和最佳实践为依据，进行建设性讨论，最终由技术负责人决策。

### Q: 审查发现太多问题怎么办？

A: 优先处理严重问题，其他问题可以作为后续改进任务。

### Q: 如何保证审查质量？

A: 使用检查清单，多人交叉审查，定期回顾审查效果。
